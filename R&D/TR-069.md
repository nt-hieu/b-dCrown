# 1. Research về protocol TR-069

Yêu cầu **“research về protocol TR-069”** nên được hiểu là: tìm hiểu TR-069 không chỉ ở mức định nghĩa, mà đủ để mô tả:

- Nó dùng để làm gì.
- Hoạt động ra sao.
- Kiến trúc gồm những thành phần nào.
- Luồng message giữa thiết bị và server như thế nào.
- Cách quản lý cấu hình, firmware, diagnostics.
- Bảo mật, rủi ro và best practices.
- Cách dựng lab mô phỏng để quan sát protocol.

TR-069, còn gọi là **CWMP — CPE WAN Management Protocol**, là giao thức do Broadband Forum phát triển để nhà mạng hoặc service provider tự động nhận diện, cấu hình, provision và bảo trì thiết bị phía khách hàng như modem/router trong suốt vòng đời thiết bị.

## 1.1. Mục tiêu research

Bài research nên trả lời được:

- TR-069 là gì?
- Vì sao ISP cần TR-069?
- Các thành phần chính gồm gì?
- ACS là gì?
- CPE là gì?
- CWMP session hoạt động như thế nào?
- Data Model dùng để làm gì?
- CPE và ACS giao tiếp với nhau ra sao?
- Các thao tác quản trị thiết bị gồm gì?
- Bảo mật của TR-069 cần lưu ý điều gì?
- TR-069 khác gì so với USP/TR-369?

# 2. Các phần cần làm trong bài research

## 2.1. Background & Motivation

Bối cảnh:

```text
ISP / Nhà mạng
    |
    | cần quản lý hàng nghìn / hàng triệu modem, router, gateway
    |
    v
ACS  <------ TR-069 / CWMP ------>  CPE tại nhà khách hàng
```

Nội dung cần phân tích:

- Trước khi có TR-069, việc cấu hình, thay đổi firmware, sửa lỗi thiết bị thường phải làm thủ công.
- Nhà mạng cần một cơ chế quản lý thiết bị tập trung.
- TR-069 giúp quản lý CPE từ xa.
- Các use case chính:
  - Auto-configuration.
  - Dynamic service provisioning.
  - Firmware/software image management.
  - Diagnostics.
  - Status/performance monitoring.

Ý nghĩa chính:

```text
TR-069 = giao thức quản trị từ xa cho thiết bị mạng phía khách hàng
```

---

## 2.2. Kiến trúc TR-069

Kiến trúc tổng quát:

```text
+--------------------+        HTTPS/SOAP/CWMP        +----------------------+
|        ACS         | <---------------------------> |         CPE          |
| Auto Config Server |                              | Modem/Router/Gateway |
+--------------------+                              +----------------------+
        |                                                       |
        | quản lý cấu hình                                     | chứa data model
        | gửi RPC commands                                     | gửi Inform
        v                                                       v
  Management System                                  Device Parameters
```

Các thành phần chính:

| Thành phần | Ý nghĩa |
|---|---|
| **ACS** | Auto Configuration Server, server trung tâm của ISP dùng để quản lý CPE |
| **CPE** | Customer Premises Equipment, thiết bị phía khách hàng như modem/router/gateway |
| **CWMP** | CPE WAN Management Protocol, protocol thực tế của TR-069 |
| **RPC Methods** | Các lệnh quản lý như lấy/sửa tham số, reboot, download firmware |
| **Data Model** | Cấu trúc tham số mà ACS dùng để hiểu và điều khiển CPE |
| **Inform** | Message CPE gửi lên ACS khi boot, định kỳ hoặc có sự kiện |

Điểm cần nhấn mạnh:

```text
TR-069 không chỉ là API.
Nó là một protocol quản trị thiết bị có session, authentication, event, RPC và data model.
```

---

## 2.3. Luồng hoạt động chính

### Flow 1: CPE boot và kết nối ACS

```text
CPE boot
  |
  v
CPE gửi Inform lên ACS
  |
  v
ACS xác thực CPE
  |
  v
ACS đọc thông tin thiết bị
  |
  v
ACS gửi cấu hình / yêu cầu cần thiết
  |
  v
CPE phản hồi kết quả
```

Ý nghĩa:

- Khi router/modem khởi động, nó chủ động gọi về ACS.
- ACS không cần biết trước thiết bị đang ở IP nào.
- ACS nhận diện thiết bị qua thông tin trong message `Inform`.

---

### Flow 2: ACS thay đổi cấu hình

```text
ACS
 |
 | SetParameterValues
 v
CPE cập nhật cấu hình
 |
 | Response
 v
ACS ghi nhận trạng thái
```

Ví dụ:

- ACS đổi SSID Wi-Fi.
- ACS đổi password Wi-Fi.
- ACS bật/tắt một service.
- ACS thay đổi Periodic Inform Interval.

---

### Flow 3: Firmware upgrade

```text
ACS yêu cầu Download firmware
        |
        v
CPE tải firmware image
        |
        v
CPE cài đặt / reboot
        |
        v
CPE báo TransferComplete về ACS
```

Ý nghĩa:

- ACS có thể yêu cầu CPE tải firmware mới.
- Sau khi hoàn tất, CPE báo lại trạng thái.
- Đây là một trong các use case quan trọng nhất của TR-069.

---

## 2.4. Data Model: TR-069, TR-181, Device tree

Data model mô tả các object/parameter mà ACS có thể đọc hoặc ghi trên CPE.

Ví dụ dạng cây:

```text
Device.
 ├── DeviceInfo.
 │   ├── Manufacturer
 │   ├── ModelName
 │   └── SoftwareVersion
 ├── ManagementServer.
 │   ├── URL
 │   ├── Username
 │   ├── PeriodicInformEnable
 │   └── PeriodicInformInterval
 ├── WiFi.
 ├── IP.
 ├── Ethernet.
 └── Diagnostics.
```

Cần research:

- Sự khác nhau giữa data model cũ `InternetGatewayDevice` và data model mới `Device:2 / TR-181`.
- Các parameter thường gặp.
- Read-only parameter.
- Writable parameter.
- Vendor-specific parameter.
- Cách ACS duyệt cây parameter.
- Cách ACS đọc/ghi parameter.

Ý nghĩa:

```text
Data Model = ngôn ngữ chung để ACS hiểu và điều khiển thiết bị CPE
```

---

## 2.5. RPC methods cần biết

Các RPC method quan trọng:

| RPC | Mục đích |
|---|---|
| `Inform` | CPE báo sự kiện/trạng thái cho ACS |
| `GetParameterValues` | ACS đọc tham số |
| `SetParameterValues` | ACS ghi/cấu hình tham số |
| `GetParameterNames` | Duyệt cây data model |
| `AddObject` | Tạo object mới |
| `DeleteObject` | Xóa object |
| `Reboot` | Khởi động lại CPE |
| `Download` | Tải firmware/file |
| `Upload` | Upload log/config |
| `FactoryReset` | Reset thiết bị |

Ví dụ:

```text
ACS muốn đổi Wi-Fi SSID
  -> dùng SetParameterValues
  -> ghi vào parameter thuộc nhánh WiFi hoặc WLANConfiguration
```

---

## 2.6. Bảo mật TR-069

TR-069 liên quan trực tiếp tới router/modem, nên bảo mật là phần bắt buộc.

Các điểm cần phân tích:

- TR-069 nên chạy qua HTTPS/TLS.
- Cần xác thực ACS và CPE.
- Không nên expose endpoint quản trị ra Internet nếu không cần.
- Cần kiểm soát Connection Request URL.
- Cần kiểm soát username/password.
- Cần kiểm soát certificate.
- Cần validate input từ ACS và CPE.

Rủi ro thực tế:

- Nếu ACS bị compromise, attacker có thể ảnh hưởng hàng loạt CPE.
- Nếu CPE implementation lỗi, có thể dẫn tới RCE, DoS hoặc lộ thông tin.
- Nếu port TR-069 bị expose sai cách, thiết bị có thể bị khai thác từ Internet.

# 3. Cách xây dựng lab mô phỏng TR-069

Có 3 mức lab.

## 3.1. Lab đơn giản nhất: GenieACS + CPE simulator

Kiến trúc:

```text
+-------------------+        CWMP / TR-069        +--------------------+
| GenieACS - ACS    | <-------------------------> | genieacs-sim - CPE |
| port 7547 CWMP    |                             | giả lập router     |
| port 7557 API     |                             +--------------------+
| port 7567 file    |
| port 3000 UI      |
+-------------------+
```

Dùng **GenieACS** làm ACS vì đây là open-source TR-069 ACS, hỗ trợ:

- Provisioning.
- Firmware upgrade.
- Khám phá parameter tree.
- Gửi task tới CPE.
- Quản lý device qua UI/API.

Các service quan trọng:

| Port | Ý nghĩa |
|---|---|
| `7547` | CWMP/TR-069 endpoint |
| `7557` | NBI API |
| `7567` | File server |
| `3000` | Web UI |

Dựng nhanh:

```bash
git clone https://github.com/GeiserX/genieacs-container.git
cd genieacs-container
docker compose --profile testing up -d
```

Sau đó mở:

```text
http://localhost:3000
```

Mục tiêu test đầu tiên:

```text
CPE simulator boot
  -> gửi Inform
  -> ACS nhận device
  -> ACS đọc parameter tree
  -> ACS gửi GetParameterValues / SetParameterValues
```

## 3.2. Lab thực tế hơn: GenieACS + OpenWrt/EasyCWMP

Kiến trúc:

```text
+-------------------+       LAN / private network       +-------------------+
| GenieACS ACS      | <-------------------------------> | OpenWrt router VM |
| Ubuntu/Docker     |                                  | EasyCWMP client   |
+-------------------+                                  +-------------------+
```

Ở mức này, CPE không còn là simulator mà là một router OpenWrt hoặc OpenWrt VM.

Cần cấu hình trên CPE:

```text
ACS URL             = http://<IP_GENIEACS>:7547
Username            = lab-cpe
Password            = lab-pass
Periodic Inform     = enable
Inform interval     = 60 hoặc 300 giây
```

Sau khi CPE gửi `Inform`, thiết bị sẽ xuất hiện trong GenieACS UI.

## 4.3. Gửi task đọc parameter

Thay `<DEVICE_ID>` bằng `_id` lấy từ API/UI:

```bash
curl -X POST "http://127.0.0.1:7557/devices/<DEVICE_ID>/tasks?connection_request" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "getParameterValues",
    "parameterNames": [
      "InternetGatewayDevice.DeviceInfo.Manufacturer",
      "InternetGatewayDevice.DeviceInfo.SoftwareVersion"
    ]
  }'
```

---

## 4.4. Gửi task refresh object

```bash
curl -X POST "http://127.0.0.1:7557/devices/<DEVICE_ID>/tasks?connection_request" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "refreshObject",
    "objectName": "InternetGatewayDevice."
  }'
```

## 4.5. Capture gói TR-069

Trên VM1:

```bash
sudo tcpdump -i any -nn -A 'tcp port 7547'
```

Khi chuyển sang HTTPS/TLS, payload sẽ bị mã hóa nên không đọc trực tiếp được bằng tcpdump.

## 5.1. Trên OpenWrt VM

Cài EasyCWMP:

```bash
opkg update
opkg install easycwmp
```

Cấu hình ACS URL:

```bash
uci set easycwmp.@acs[0].url='http://192.168.100.10:7547/'
uci set easycwmp.@acs[0].username='cpe1'
uci set easycwmp.@acs[0].password='cpe1'
uci set easycwmp.@acs[0].periodic_enable='1'
uci set easycwmp.@acs[0].periodic_interval='60'
uci commit easycwmp
/etc/init.d/easycwmp restart
```

Mục tiêu:

